# 삼탄 TMS 사용설명서

**TECHKING ETCRANE 385/95R24(14.00R24) · Road Train Tire Monitoring System**
PT. Trasindo Murni Perkasa · Kalimantan Field

---

## 1. 개요

삼탄 TMS(Tire Monitoring System)는 칼리만탄 현장에서 운용 중인 **Road Train 차량의 TECHKING ETCRANE 385/95R24(14.00R24) 시험 타이어**를 모니터링하기 위한 웹 대시보드입니다. 차량별 공기압·트레드(마모) 추이를 추적하고, 교체·폐기 이력과 타이어 수명 데이터베이스를 한 화면에서 관리합니다.

| 항목 | 내용 |
|------|------|
| 대상 타이어 | TECHKING ETCRANE 385/95R24(14.00R24) (Tube Type) |
| 신품 트레드 | 22mm |
| 모니터링 차량 | 5대 (CH 834 / 839 / 832 / 835 / 849) |
| 시험 타이어 수 | 차량당 10본 (Head/프라임무버) = 총 50본 |
| 보고 주기 | 약 2주 간격 회차 점검 |

> **참고:** Road Train은 Head(프라임무버) + Trailer 1 + Dolly + Trailer 2로 구성됩니다. 이 중 **Head의 10본(385/95R24(14.00R24))만 시험 대상**이며, Vessel·Dolly의 12.00R24 타이어는 모니터링에서 제외됩니다.

### 공통 영역 설명

- **헤더 (상단 고정):** 좌측에 로고, 가운데 시스템명("Tire Monitoring System (TMS)"), 우측에 현재 탭 제목·설명과 **언어 토글(KO/ID)** 이 표시됩니다. 스크롤해도 상단에 고정됩니다.
- **언어 전환 (KO / ID):** 헤더 우측의 토글로 **한국어 ⇄ 인도네시아어(Bahasa)** 를 전환합니다. 모든 탭의 화면 문구(라벨·표 헤더·버튼·모달·안내문)가 즉시 바뀌며, 선택한 언어는 브라우저에 저장되어 다음 접속 때도 유지됩니다. (기본값: 한국어. 시리얼·수치·코드 등 데이터 값은 공통)
- **탭 내비게이션:** 8개 탭을 클릭해 화면을 전환합니다. 현재 탭은 네이비 밑줄로 강조됩니다. 모바일 화면에서는 영문 약칭(Dashboard/Layout/Load/Replacement/Lifetime DB/Input/Trend/Pressure)으로 표시됩니다.
- **푸터:** 회사·현장 정보 및 시스템 표기(`Copyright © ASEOA`).

## 3. 탭별 기능 및 사용법

### 3.1 대시보드 (Dashboard)

전체 시험 현황을 한눈에 요약하는 첫 화면입니다.

**① 시험 현황 요약 (KPI 카드)**
8개의 카드로 핵심 지표를 표시합니다.

| 카드 | 의미 |
|------|------|
| 모니터링 차량 | 시험 대상 차량 대수 |
| 시험 타이어 | 차량 수 × 10본 (Head 기준) |
| 정상 / 주의 / 교체권장·폐기 | 상태 판정에 따른 타이어 본수 (색상으로 구분) |
| 누적 폐기 | 폐기(Scrap) 처리된 타이어 수 |
| 장착 (DB) / 예비 타이어 | Lifetime DB 기준 장착·예비 본수 |

**② 차량(CH)별 현황 테이블**
차량마다 최초 장착일(첫 점검 회차), 최근 점검일, 주행 HM, 주행 KM, 정상/주의/교체권장 본수, 교체 이력 횟수를 행으로 보여줍니다. **주행 HM·주행 KM은 시험 기간 실제 주행량(최신 회차 − 최초 회차)** 입니다.

- **하단 합계·평균 행:** 주행 HM·KM은 **평균**(값이 있는 차량 기준), 정상·주의·교체권장·교체이력은 **합계**로 표시됩니다.
- **주의·교체권장 숫자 클릭 → 알림 모달:** 해당 차량의 주의(마모 진행) 또는 교체권장/위험(트레드·저압·폐기 사유) 항목을 모달로 상세 표시합니다. (`tmsUtils.getUnitAlerts()`에서 레벨별 필터)
- 표 하단에 트레드/공기압 판정 기준이 안내됩니다 (신품 22mm · 주의 ≤8mm · 교체권장 ≤5mm | 조향축 100psi · 구동축 116psi).

> 이전의 항상 노출되던 "주의·교체 알림" 목록 섹션은 제거되고, 위 **차량별 현황 표의 주의·교체권장 클릭 모달**로 통합되었습니다.

**③ 결재 라인**
Admin Tire → Foreman → Manager → PJO → Pres. Director 순의 결재자를 표시합니다.

### 3.2 차량 배치도 (Layout)

Road Train의 타이어 배치를 시각적으로 보여주고, 개별 타이어를 클릭해 상세 정보를 확인하는 화면입니다.

**사용 순서**
1. 상단의 **차량(CH) 버튼**(834/839/832/835/849)을 눌러 대상 차량을 선택합니다.
2. **Road Train 배치도**가 진행 방향에 맞춰 **1.5배 확대·가운데 정렬**로 표시됩니다. (폭이 넘치면 가로 스크롤). 배경에는 실제 Road Train 차량 사진이 흐릿하게(투명도 0.25) 깔립니다.
   - **Head(파란 테두리):** 시험 대상 10본. 색상으로 상태를 표시하며 **클릭 가능**합니다.
   - **Trailer 1 / Dolly / Trailer 2(빨간 테두리):** 12.00R24, 시험 외(점선·반투명 번호 셀로 표시, 클릭 불가).
3. **범례**로 색상별 상태를 확인합니다. (정상=초록 / 주의=노랑 / 교체권장=빨강 / 시험 외=점선)

**타이어 상세 모달**
Head 타이어를 클릭하면 팝업이 열리며 다음 정보를 제공합니다.
- 포지션 번호·코드(예: #1 · L1)와 상태 배지
- 권장 공기압, 시리얼(교체 시 `구→신` 표기)
- 최근 공기압 / 최근 트레드 (측정일 포함)
- 비고 및 폐기 정보(있을 경우)
- **트레드 추이** / **공기압 추이** 미니 차트 (기준선 포함)

> 포지션 코드: L1·R1(조향축), L2/L3·R2/R3은 In/Out 듀얼 타이어입니다. 내측(In) 듀얼은 측정 제외(N/A)인 경우가 있습니다.

> **시리얼 바로가기:** 모달의 **시리얼을 클릭**하면 "타이어 수명 DB" 탭으로 이동해 해당 시리얼 행으로 스크롤·하이라이트됩니다. (압력·트레드 추이 표, 타이어 교체 이력 표의 시리얼도 동일)

### 3.3 하중계산 (Load)

Road Train 유닛별 **축하중과 정격 대비 부하율**을 계산해 과적·임계 상태를 확인하는 화면입니다.

**① 안내 배너**
총중량(kg)과 정격 하중 기준(Head 6,000 / Vessel·Dolly 4,750 kg/본)을 안내합니다.

**② 임계 경고**
부하율 100% 이상(정격 초과) 유닛이 있으면 빨간 경고 배너로 해당 유닛·부하율을 표시합니다. (예: Trailer 1)

**③ 유닛별 하중 현황 표**
유닛(Head/Trailer 1/Dolly/Trailer 2)별 규격·본수·공차(kg)·적재(kg)·총중량·분배율·**타이어당 하중**·정격·상태를 보여주며, 하단에 합계 행을 제공합니다.

**④ 정격 대비 부하율 막대**
유닛별 부하율을 막대로 시각화합니다. 정상 <90%(초록) · 주의 90~100%(노랑) · 초과 ≥100%(빨강), 점선은 정격 100% 기준선입니다.

> **계산식:** 타이어당 하중 = 총중량 × 분배율 ÷ 본수 (`tmsUtils.loadPerTire`), 부하율 = 타이어당 하중 ÷ 정격 (`tmsUtils.loadRatio`).
> **정격 근거:** 모니터링 시트 + Techking OTR Databook (Head 385/95R24 6,000kg / Vessel·Dolly 12.00R24 4,750kg, 본당).
> **공차·적재값**은 예시이며, 실측 적재 시트 입력 시 자동 갱신됩니다 (`src/data/loadData.ts · LOAD_UNITS`).

### 3.4 타이어 교체 이력 (Replacement)

교체가 실제 있었던 (차량·포지션)을 **체인으로 묶어** 표시하는 화면입니다.

- 열 구성: 차량 · 포지션 · **1차 장착일 · 1차 시리얼**(원본) · **2차 장착일 · 2차 시리얼 · 2차 교체사유**(1차가 내려간 사유 — 폐기면 손상코드, 아니면 "로테이션").
- 같은 포지션이 **두 번 이상 교체된 경우(3차 장착 존재)**, **3차 장착일 · 3차 시리얼 · 3차 교체사유** 열이 자동으로 추가됩니다.
- **차량(CH)** 클릭 → 차량 배치도, **시리얼** 클릭 → 타이어 수명 DB(해당 행 포커스), **손상코드** 클릭 → 손상 코드 모달로 이동합니다.

**손상 코드 (모달)**
폐기 사유에 사용되는 코드와 설명입니다. 교체사유의 **손상 코드(예: `CD (Crown Damage)`)를 클릭**하면 모달이 열리며 해당 코드가 강조 표시됩니다.

| 코드 | 의미 | 코드 | 의미 |
|------|------|------|------|
| WO | Worn Out | ILB | Inner Liner Bulge |
| SC | Sidewall Cut | CD | Crown Damage |
| BC | Bead Cut | SWB | Spot Wear Brake |
| BB | Bead Bulge | IW | Irregular Wear |
| SHC | Shoulder Cut | SCF | Sidewall Scuffing |
| BO | Burn Out | RF | Run Flat |
| ORF | Over Run Flat | OTH | Other |

> 손상 코드 모달은 타이어 교체 이력 탭과 타이어 수명 DB 탭(비고의 손상 코드)에서 공통으로 사용됩니다.

### 3.5 타이어 수명 DB (Lifetime DB)

전체 타이어 목록(No, 시리얼, 최초 장착일, 최근 트레드, 주행 KM, 주행 HR, 예상 KM, 예상 HM, 상태, 비고)을 보여주는 화면입니다.

- **제목행 고정:** 표가 길어(전체 70본) 세로 스크롤 시 **헤더 행이 상단에 고정**됩니다.
- **행을 클릭**하면 해당 타이어의 **전체 히스토리 모달**이 열립니다 — 최초 장착일·재장착일·탈착일·폐기일·손상코드·HM Reset 여부·주행/예상 수명·최근 트레드 + **장착 이력(위치·차량 변경)** + 측정 회차별(점검일·트레드·KM·HM) 이력. (`tmsUtils.tireHistory()`, `tmsUtils.tireMovements()`)
- **장착 이력(위치·차량 변경)**: 이 타이어가 거쳐 온 **장착일·차량(CH)·포지션·구분(최초/교체 장착)** 을 시간순으로 보여줍니다. 현재 장착 위치는 강조 표시됩니다. 로테이션(다른 차량·포지션으로 이동)된 타이어는 이 표에 여러 행으로 나타나 위치·차량 변경 이력을 확인할 수 있습니다. (원본 장착 + `replacements`의 재장착 이벤트를 합쳐 재구성)
- **최초 장착일**은 원본 Lifetime 시트의 입고·조립일(Assembly Date) 기준으로 입력되어 있습니다 (`0003_install_dates.sql`). **폐기일**은 행 클릭 모달에서 확인합니다.
- **최근 트레드**: 해당 타이어의 가장 최근 측정값(mm). ≤8mm 노랑·≤5mm 빨강. (`tmsUtils.latestTreadOf()`)
- **주행 KM·주행 HR**: 시리얼로 찾은 포지션의 트레드 측정 회차에서 (최신−최초) 주행거리/가동시간. **예상 KM·예상 HM**: 트레드 vs 누적 주행거리/가동시간 선형회귀로 0mm 도달 시점을 외삽한 추정값. (`tmsUtils.lifeCalc()`)
- **목표 수명 75,000 km** — 예상수명이 목표 이상이면 초록색으로 표시됩니다.
- km 데이터가 2회차 미만이거나 변동이 없으면 `−`로 표시됩니다. (예: HM Reset·주행거리 미기록 차량)
- 교체 타이어는 교체일 기준으로 해당 타이어 장착 구간만 계산하며, 신품(22mm) 기준 회차가 없으면 `−`로 표시됩니다.
- **예상수명**은 `Scrap`·`Not Install` 상태 타이어에는 표시하지 않습니다(`−`).
- **비고**에는 손상 코드(폐기 사유)·기존 노트·HM Reset(CH 832·835 장착 타이어)이 함께 표시됩니다. 손상 코드 클릭 시 손상 코드 모달이 열립니다.
- **`측정 외` 배지**: 모니터링 5대(834·839·832·835·849) 외 차량에 장착되었거나 예비 타이어인 경우 표시되며, 행이 옅게 흐려집니다. 점검 데이터가 없어 트레드·주행·예상이 `−`로 표시되는 것은 정상입니다.
- 우측 **상태 필터** 드롭다운으로 `전체 / Install / Scrap / Spare / Not Install` 을 골라 볼 수 있습니다. 상태는 색상 배지로 구분: Install(초록), Scrap(빨강), Spare(노랑), Not Install(회색). 필터 결과 본수 / 전체 본수가 함께 표시됩니다.
- 하단 **평균 행**: 표시 행 기준 최근 트레드·주행 KM·HR·예상 KM·HM의 평균을 보여줍니다.
- **엑셀 다운로드**: 우측 상단 **[엑셀 다운로드]** 버튼으로 **현재 필터에 표시된 행**을 `.xlsx`로 내보냅니다(No·시리얼·최초 장착일·최근 트레드·주행 KM·HR·예상 KM·HM·상태·비고). 파일명 `TireLifetimeDB_YYYY-MM-DD.xlsx`. (엑셀 라이브러리는 클릭 시 동적 로드되어 초기 로딩에 영향 없음)

> **HM Reset이란?** HM(Hour Meter)은 차량의 누적 가동시간계입니다. 엔진/계기 교체 등으로 **운용 중 가동시간계·주행거리가 0으로 초기화**된 차량을 가리킵니다(원본 기준 CH 832·835). 리셋 이전 누적 기록이 끊기므로 해당 차량 타이어의 **수명·예상수명은 `−`로 표시**될 수 있습니다. (DB: `vehicles.hm_reset`, `0004_vehicle_hm_reset.sql`)

> 다른 탭(차량 배치도·압력추이·교체 이력)에서 **시리얼을 클릭**하면 이 탭으로 이동하며 해당 행이 스크롤·하이라이트됩니다(필터 자동 해제).

### 3.6 점검 입력 (Input)

각 차량의 회차 점검결과(공기압·트레드)를 직접 입력하고 **Supabase 데이터베이스**에 저장하는 화면입니다. 현장 관리자가 측정값을 바로 기록할 수 있습니다.

> **🔒 로그인 필요:** 점검 입력(저장·삭제)은 **로그인한 사용자만** 가능합니다(RLS `0006`). 비로그인 시 이 탭에 **로그인 화면**(이메일·비밀번호)이 표시되며, 로그인하면 입력 폼이 나타납니다. 로그인 상태는 헤더 우측에 이메일·**로그아웃** 버튼으로 표시되고 브라우저에 유지됩니다. 다른 탭(조회)은 로그인 없이 사용 가능합니다.
> 계정은 Supabase 대시보드 **Authentication → Users**에서 생성합니다. (이메일/비밀번호)

**① 차량 선택**
상단 CH 버튼으로 입력 대상 차량을 선택합니다. 선택 시 해당 차량의 10개 포지션·시리얼이 폼에 자동 채워집니다.

**② 회차 메타 입력**
- **점검일** (기본값: 오늘)
- **HM** (가동시간)
- **KM** (주행거리)

**③ 포지션별 입력**
10개 포지션(L1~R3 Out)마다 **공기압(psi)**, **트레드(mm)**, **시리얼**을 입력합니다. 권장 공기압이 함께 표시됩니다.
- 내측 듀얼(In) 타이어는 미측정 시 비워두면 N/A로 처리됩니다.
- 공기압·트레드 중 1개 이상 입력된 포지션만 저장됩니다.

**④ 저장**
[저장] 버튼으로 Supabase에 기록합니다. 동일 `(차량 · 점검일 · 포지션)` 조합은 **자동 갱신(upsert)** 되어 중복이 생기지 않습니다. 저장 성공 시 토스트 알림이 표시됩니다.

**⑤ 저장된 점검 이력**
하단에 선택 차량의 저장된 회차가 **점검일 최신순**으로 표시됩니다. 회차별로 HM·KM과 포지션별 값을 보여주며, 휴지통 아이콘으로 회차 전체를 삭제할 수 있습니다. [새로고침]으로 최신 데이터를 다시 불러옵니다.

> **데이터 저장 위치:** Supabase `public.inspections` 테이블. 자세한 내용은 [7. Supabase 연동](#7-supabase-연동-개발자-참고) 참조.

> **차트 자동 반영:** 모든 탭의 데이터는 Supabase에서 로드됩니다. 저장한 점검값은 **대시보드 · 차량 배치도 · 압력·트레드 추이 · 타이어 공기압관리** 탭의 차트·표에 새 회차로 자동 반영됩니다.

### 3.7 압력·트레드 추이 (Trend)

선택한 차량의 회차별 측정값을 선 그래프와 표로 추적하는 화면입니다.

**사용 순서**
1. **차량(CH)** 버튼으로 대상 차량 선택.
2. 우측 **항목 선택** 드롭다운에서 보고 싶은 지표 선택:
   - **트레드 (mm)** — 기준선: 주의 8mm, 교체 5mm 표시
   - **공기압 (psi)** — 권장압(100/116psi) 기준선 표시

**① 추이 차트**
10개 포지션(L1~R3 Out)을 색상별 선으로 그려 회차에 따른 변화를 보여줍니다. 마우스를 올리면 값이 툴팁으로 표시됩니다.

**② 회차별 상세 데이터 테이블**
포지션별로 각 점검 회차의 측정값을 셀로 보여줍니다.
- 트레드 값은 색상으로 상태를 구분(초록/노랑/빨강).
- `N/A`는 내측 듀얼 타이어(측정 제외), `−`는 데이터 없음.
- 우측 끝에 미니 추이선(스파크라인) 표시.
- 하단에 회차별 **HM / KM** 행 제공.
- 시리얼 클릭 → 타이어 수명 DB로 이동.

> 트레드 값은 타이어의 4개 그루브 중 **최소값** 기준입니다.

### 3.8 타이어 공기압관리 (Pressure)

현장 관리자(Pengawas)가 공기압 점검 기준·절차·대응을 참조하는 화면입니다. 점검 데이터를 단순 조회하는 "압력·트레드 추이" 탭과 달리, **현장 작업 가이드** 중심으로 구성되며 차량 데이터에서 이상값을 자동 계산해 함께 보여줍니다. (한국어 + 인도네시아어 병기)

**① 안내 배너**
관리자용 공기압 점검 페이지임을 안내합니다. 회차 점검 시 기준에 따라 측정·기록하고 이상 발견 시 감독자에게 보고하도록 안내합니다.

**② 권장 공기압 기준 (Standar Tekanan)**
- **조향축(Steering, L1·R1):** 100 psi
- **구동축(Drive, L2·L3·R2·R3):** 116 psi
- TECHKING ETCrane 시험 목표압 **130 psi** 상향 검토 중. 내측 듀얼(In) 타이어는 측정 제외(N/A).

**③ 최근 공기압 점검 현황 (실시간 계산)**
각 차량의 최근 회차 실측값에서 자동 계산한 지표를 보여줍니다.
- KPI 카드: 측정 본수 / 정상 / 편차 주의 / 저압 위험
- **공기압 이상 알림** 목록 — `CH · 포지션 · 실측 psi (권장 대비 편차%)` 형식. 위험 항목이 상단 우선 정렬.

> 판정: **저압** = 권장의 85% 미만 · **편차** = 권장 대비 ±15% 초과

**④ 상태 판정 기준 (Kriteria Penilaian)**
- **정상:** 권장 대비 ±5% 이내 — 조치 불필요
- **주의:** 권장 대비 ±5~15% 편차 — 재측정·보충 권장
- **위험:** 저압 85% 미만 또는 과압 +15% 초과 — 운행 전 즉시 조치

**⑤ 공기압 점검 절차 (Prosedur)**
체크 가능한 7단계 절차 목록입니다. (체크 상태는 화면 표시용으로, 새로고침 시 초기화)
냉간 측정 → 포지션별 실측 기록 → 내측 듀얼 N/A 처리 → 게이지 정확도 확인 → 컴프레서 130psi 확인 → 이상 시 감독자 보고 → OTR Data Tracking Sheet 기록.

**⑥ 측정 장비 관리 (Peralatan Ukur)**
컴프레서 최대압(130 psi 도달 필요), 게이지 정확도(±5% 이내, 정기 캘리브레이션), 현재 운용압(125 psi)을 안내합니다.

**⑦ 이상 징후 및 대응 (Gejala & Tindakan)**
증상 · 원인 · 위험 · 조치를 표로 제공합니다. 저압 / 과압 / 편마모(TOE IN) / 좌우 편차.

**⑧ 점검 담당 라인**
보고 체계(Admin Tire · Foreman · Manager · PJO)와 점검 주기(약 2주)를 안내합니다.

## 7. Supabase 연동 (개발자 참고)

이 앱의 **모든 데이터는 Supabase에 저장**되며, 앱 시작 시 전부 로드됩니다. (정적 하드코딩 데이터 없음)

**로드 동작 (`src/data/loadData.ts`):** 앱 시작 시(및 점검 저장/삭제 시) `loadData()`가 전체 테이블을 병렬로 불러와 `TMS_DATA`를 구성합니다.
- 참조 데이터(스펙·차량·포지션·교체이력·수명DB·손상코드) + 측정 데이터(inspections)를 합쳐 회차 배열(dates·hm·km·포지션별 공기압·트레드)을 만듭니다.
- 입력 점검일이 기존 회차와 같으면 **덮어쓰기(overlay)**, 새 날짜면 **새 회차로 추가**됩니다.
- `summary`(상태별 본수)는 `lifetimes`에서 자동 집계합니다.
- 대시보드/추이/배치도/공기압관리 탭은 `key={dataVersion}`로 데이터 갱신 시 재렌더됩니다. 로드 완료 전에는 로딩 화면이 표시됩니다.

### 7.1 연결 설정

연결 정보는 환경변수로 주입하며, 누락 시 `src/lib/supabase.ts`의 기본값으로 폴백합니다.

```bash
# .env  (git 추적 제외 — .env.example 참고)
VITE_SUPABASE_URL=https://vndbzanhoumhhuneufni.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_xxxxxxxxxxxx
```

> `sb_publishable_…` 키는 브라우저 노출용 **공개 키**이므로 클라이언트에 포함되어도 안전합니다.
> 이 프로젝트는 신규 API 키 체계를 사용하므로 레거시 JWT(anon) 키가 아닌 **publishable 키**를 써야 합니다.

### 7.2 데이터베이스 스키마

모든 테이블은 **RLS 활성화** 상태입니다. 보안 강화(`0006`) 적용 후 정책은 **읽기(SELECT)는 anon 포함 모두 허용 / 쓰기(INSERT·UPDATE·DELETE)는 인증 사용자(authenticated)만 허용**입니다.

> ⚠️ **점검 입력 영향:** `0006` 적용 후에는 점검 입력 저장·삭제에 **Supabase Auth 로그인(authenticated 세션)** 이 필요합니다. 로그인 없이 anon 키만으로는 조회만 가능합니다. (적용 전에는 anon 쓰기 허용)

**측정 데이터 — `inspections`** (`0001_inspections.sql`) · 1행 = (차량·점검일·포지션):

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | uuid | PK |
| `ch` | text | 차량 번호 |
| `inspection_date` | date | 점검일 |
| `hm` / `km` | numeric | 가동시간 / 주행거리 |
| `driver` | text | 차량 운전자 (선택 입력 · `0008`) |
| `pos` | text | 포지션 (L1, L2 In …) |
| `serial` | text | 타이어 시리얼 |
| `pressure` / `tread` | numeric | 공기압 psi / 트레드 mm (null = N/A) |
| `remark` | text | 비고 |
| `created_at` / `updated_at` | timestamptz | 생성/수정 시각 |

> 유니크 제약 `(ch, inspection_date, pos)` → 동일 조합 재저장 시 **upsert(갱신)**.
>
> `hm` · `km` · `driver`는 **회차 메타**이므로 같은 (차량·점검일)의 포지션 행마다 동일 값이 중복 저장됩니다. 조회 시 회차의 첫 행(`rows[0]`) 기준으로 읽습니다.

**참조 데이터** (`0002_reference_tables.sql`):

| 테이블 | 내용 | 주요 컬럼 |
|--------|------|-----------|
| `tire_config` | 타이어 스펙 (전역 1행) | brand, type, size, tubetype, new_tread, capacity |
| `vehicles` | 차량 목록 | ch(PK), seq |
| `vehicle_positions` | 차량별 포지션·시리얼 | ch, pos, serial, serial2, remark, seq |
| `replacements` | 교체 이력 | ch, pos, date, new_serial |
| `lifetimes` | Lifetime DB | no(PK), serial, scrap, damage, install, lifetime, status, note |
| `damage_codes` | 손상 코드 | code(PK), description, seq |

> `summary`(상태별 본수)는 별도 저장하지 않고 `lifetimes`에서 로드 시 집계합니다.

### 7.3 스키마 적용 방법

신규 환경에서는 마이그레이션 2개를 순서대로 1회 적용합니다.

- **방법 A — Supabase 대시보드:** SQL Editor에 `0001_inspections.sql` → `0002_reference_tables.sql` 순으로 붙여넣고 실행.
- **방법 B — psql:**
  ```bash
  for f in supabase/migrations/0001_inspections.sql supabase/migrations/0002_reference_tables.sql; do
    PGPASSWORD='<DB비밀번호>' psql \
      -h aws-1-ap-southeast-1.pooler.supabase.com -p 5432 \
      -U postgres.vndbzanhoumhhuneufni -d postgres -f "$f"
  done
  ```
  (초기 데이터 시드는 별도 진행 — 현재 운영 DB에는 이미 이관 완료됨)

## 8. Git 커밋 및 배포 (개발자 참고)

### 8.1 브랜치 구조

저장소(`hungaseo-arch/samtan`)는 **public**이며 브랜치 2개를 용도별로 나눠 씁니다.

| 브랜치 | 내용 | 편집 방법 |
|--------|------|-----------|
| `main` | 소스 코드 (`src/`, `supabase/`, 설정 파일) | 직접 커밋 |
| `gh-pages` | 빌드 산출물 (`index.html`, `assets/`) | **직접 편집 금지** — 배포 명령이 자동 생성 |

GitHub Pages는 정적 파일만 서빙하므로 React·TypeScript 소스를 그대로 올릴 수 없고, 컴파일된 결과물이 별도로 필요합니다. `gh-pages` 브랜치가 그 역할이며 커밋 메시지가 전부 `Updates`인 이유도 사람이 아닌 `gh-pages` 패키지가 자동 생성하기 때문입니다.

### 8.2 커밋 절차

```bash
git status                      # 변경 파일 확인 (의도치 않은 파일이 섞였는지 먼저 확인)
git add <파일>                   # 파일을 지정해 스테이징 (git add . 는 지양)
git commit -m "<메시지>"
git push origin main
```

정상 업로드 여부는 아래로 확인합니다. 로컬 참조가 오래됐을 수 있으므로 `fetch` 후 비교합니다.

```bash
git fetch origin
git status -sb                  # "## main...origin/main" 뒤에 ahead/behind 표시가 없으면 동기화 완료
git log origin/main..HEAD --oneline   # 출력이 없으면 미푸시 커밋 없음
```

### 8.3 커밋 메시지 규칙

현재 저장소의 관행은 **영문 한 줄 요약**이며, 범위를 앞에 붙이고 부연은 괄호로 답니다. 형식은 `<범위>: <변경 내용> (<부연>)`.

```
PressureTab row2: equal column heights (현황 · 판정)
Fix 0007: rename current_role()->app_role() (reserved word) and apply
Remove header login button (login via inspection tab button/modal)
```

- 범위는 컴포넌트명(`InspectionTab`, `Header`)이나 마이그레이션 번호(`Fix 0007`)를 씁니다.
- "무엇을 했는지"를 명령형·명사구로 적고, 마침표는 생략합니다.
- 배경 설명이 필요하면 한 줄 비우고 본문에 덧붙입니다.

### 8.4 배포

```bash
npm run deploy
```

이 한 줄이 빌드부터 푸시까지 수행합니다 (`package.json`의 `deploy` 스크립트).

1. `VITE_BASE=/samtan/`로 빌드 → `dist/` 생성
2. `dist/index.html`을 `404.html`로 복사 (SPA 라우팅용 — 하위 경로 직접 접속 대응)
3. `gh-pages -d dist` → `dist` 내용을 `gh-pages` 브랜치에 커밋·푸시

배포 결과: <https://hungaseo-arch.github.io/samtan>

> ⚠️ **GitHub Actions 자동 배포가 아닙니다.** `main`에 푸시해도 사이트는 바뀌지 않으며, 반드시 로컬에서 `npm run deploy`를 직접 실행해야 반영됩니다.

### 8.5 자격증명 주의

`gh-pages` 브랜치는 public 저장소에 그대로 공개되므로 **`dist/`에 들어간 것은 전부 인터넷에 게시된다고 보아야 합니다.**

- `.env`는 `.gitignore` 대상이며 커밋하지 않습니다. 예시는 `.env.example`로 관리합니다.
- `sb_publishable_…` 키는 브라우저 노출용 공개 키이므로 클라이언트 번들에 포함되어도 안전합니다. 반면 **DB 비밀번호·`service_role` 키는 절대 커밋 금지**입니다.
- `PGPASSWORD` 등은 명령에 하드코딩하지 말고 셸 환경변수로 주입합니다.
- 에이전트 설정(`.claude/`)은 permission 규칙에 명령 전문이 그대로 저장되어 자격증명이 섞이기 쉬우므로 `.gitignore` 대상입니다.

> **알려진 함정 —** `gh-pages` 패키지는 기존 파일을 정리할 때 dotfile(`.`으로 시작하는 파일)을 삭제 대상에서 제외합니다. 따라서 한 번 배포된 dotfile은 이후 재배포로도 사라지지 않고 계속 남습니다. 제거하려면 `gh-pages` 브랜치에서 직접 `git rm` 후 푸시해야 합니다.
>
> 실제로 과거 배포 잔재인 `.claude/settings.json`이 DB 비밀번호를 담은 채 공개되어 있었고, 비밀번호 교체 + 파일 제거로 조치했습니다. **자격증명이 노출되면 파일 삭제만으로는 부족하며 반드시 해당 키를 교체(rotate)해야 합니다.** 커밋 히스토리와 외부 캐시에는 계속 남기 때문입니다.
