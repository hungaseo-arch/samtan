# 과제 — 커스텀 SMTP 연결 (대량 계정 발급 대비)

- **상태:** 미착수 (필요해지면 진행)
- **작성일:** 2026-07-22
- **관련 문서:** [사용설명서](사용설명서.md) 7.2 권한 등급 · 계정 발급 방식

---

## 1. 왜 지금은 필요 없는가

현재는 **관리자가 계정을 직접 생성**하고 초기 비밀번호(`Ascendo123`)를 당사자에게 구두·메신저로 전달합니다. 메일을 전혀 쓰지 않으므로 SMTP 없이 동작합니다.

이 방식을 택한 이유는 초대 메일이 실제로 실패했기 때문입니다. Supabase 기본 SMTP는 발송량이 매우 제한적이고(시간당 소량), 링크 만료·복귀 주소 오류가 겹치면서 초대받은 사람이 로그인할 수 없었습니다.

## 2. 언제 이 과제가 필요한가

아래 중 하나라도 해당되면 착수합니다.

- **한 번에 여러 명**(대략 10명 이상)에게 계정을 발급해야 할 때 — 비밀번호를 개별 전달하는 부담이 커집니다.
- 현장 인원이 늘어 **비밀번호 분실 재설정 요청이 잦아질 때** — 현재는 관리자가 계정을 삭제 후 재생성해야 합니다.
- 공통 초기 비밀번호를 쓰는 것이 **보안상 부담**이 될 때 — 인원이 많아질수록 첫 로그인 전 계정이 노출될 위험이 커집니다.

## 3. 해야 할 일

### 3.1 자체 도메인 확보

메일 발신 주소로 쓸 도메인이 필요합니다. `noreply@<회사도메인>` 형태를 권장합니다.

- 기존 회사 도메인(`ptascendo.com` 등)을 쓸 수 있는지 먼저 확인 — 별도 구매가 불필요할 수 있습니다.
- DNS 레코드를 직접 추가할 수 있는 권한이 필요합니다(아래 SPF·DKIM 설정에 필수).

### 3.2 Resend 설정

[resend.com](https://resend.com) 을 발송 서비스로 사용합니다. 무료 구간이 있고 SMTP 연동이 간단합니다.

1. Resend 가입 → **Domains**에 위 도메인 추가
2. 안내되는 **DNS 레코드(SPF·DKIM·DMARC)를 도메인에 등록** — 이걸 빠뜨리면 메일이 스팸으로 가거나 아예 거부됩니다
3. 도메인 인증(Verified) 확인
4. **API Keys**에서 발송용 키 발급

### 3.3 Supabase에 SMTP 연결

대시보드 → **Project Settings → Authentication → SMTP Settings**에서 Custom SMTP를 켜고 입력합니다.

| 항목 | 값 |
|------|-----|
| Host | `smtp.resend.com` |
| Port | `465` (SSL) 또는 `587` (STARTTLS) |
| Username | `resend` |
| Password | Resend API 키 |
| Sender email | `noreply@<도메인>` |
| Sender name | `Samtan TMS` |

> ⚠️ API 키는 **대시보드에만 입력**합니다. 저장소·클라이언트 코드에 넣지 마세요. (과거 gh-pages에 자격증명이 공개된 사례가 있습니다 — 사용설명서 8.5 참고)

### 3.4 앱 코드 변경

SMTP가 붙으면 초대 방식을 되살릴 수 있습니다. 현재 코드는 직접 생성 방식만 지원합니다.

- `supabase/functions/admin-users/index.ts` 에 `invite` 액션 추가
  → `admin.auth.admin.inviteUserByEmail(email, { redirectTo })`
  → 생성 직후 `updateUserById` 로 역할 부여 (invite 호출은 `app_metadata` 를 받지 않음)
- `src/api/adminUsers.ts` · `src/components/AdminUsersModal.tsx` 에 초대 경로 추가
  (직접 생성 방식은 **남겨둘 것** — 메일이 막힌 현장에서 여전히 유용)
- **Redirect URLs 허용목록**에 앱 주소가 등록되어 있어야 링크가 앱으로 돌아옵니다
  `https://hungaseo-arch.github.io/samtan/**` · `http://localhost:8080/**`

이력 참고: 초대 구현은 커밋 `3047ab4`·`68a597a` 에 있었고 `0e6861d` 에서 직접 생성으로 교체되었습니다. 되살릴 때 참고할 수 있습니다.

## 4. 검증 항목

- [ ] Resend 도메인 상태가 **Verified**
- [ ] 실제 초대 메일이 **받은편지함**에 도착(스팸함 아님)
- [ ] 링크 클릭 → 앱의 비밀번호 설정 화면으로 이동
- [ ] 비밀번호 설정 후 정상 로그인, 부여한 역할이 반영됨
- [ ] 10건 연속 발송 시 지연·차단 없음
