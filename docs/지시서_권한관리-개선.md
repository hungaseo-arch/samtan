# 작업 지시서 — 권한관리(RBAC) 개선

> **대상**: VSCode / Claude Code (이 리포지토리에서 직접 작업)
> **프로젝트**: samtan_tms (Vite · React 19 · TS · Tailwind · shadcn-ui · Supabase)
> **작성일**: 2026-07-22
> **범위**: 권한관리(Role-Based Access Control / RBAC / Kontrol Akses Berbasis Peran) 계층의 개선만. 데이터 스키마·차트 로직은 건드리지 않음.

---

## 0. 현행 구현 요약 (그대로 유지할 부분)

이미 견고하게 구현되어 있으므로 **재작성하지 말 것**. 아래를 기준(baseline)으로 개선만 얹는다.

| 계층 | 파일 | 현재 상태 |
|---|---|---|
| DB 정책 | `supabase/migrations/0007_rls_role_based.sql` | 역할기반 RLS. `admin`=전체 / `staff`=insert·update / `user`(기본)=읽기. DELETE는 `admin`만. |
| 역할 저장 | `auth.users.app_metadata.role` (JWT 포함) | Supabase 대시보드에서 `{"role":"admin"|"staff"}` 수동 설정. 미지정=`user`. |
| 프런트 컨텍스트 | `src/auth/AuthProvider.tsx` | `role`, `canWrite`(staff·admin), `canDelete`(admin), `canDownload`(staff·admin), `isAdmin` 노출. |
| 로그인 | `src/auth/LoginCard.tsx`, `src/pages/home/Index.tsx` | 이메일/비밀번호 로그인 모달. |
| 게이팅 적용됨 | `InspectionTab.tsx`(입력·저장=canWrite, 삭제=canDelete), `LifeTab.tsx`(엑셀=canDownload) | 정상 |

**권한 판정 원칙(유지)**: 계층 누적형 — `admin ⊇ staff ⊇ user`. 삭제는 admin 전용. 방어선은 **RLS가 최종**, 프런트 게이팅은 UX 보조.

## 1. 개선 과제 (우선순위 순)

각 과제는 **파일 · 변경 내용 · 수용 기준(Acceptance)** 형식. 위에서부터 순서대로 처리.

### [P0-1] RLS 권한거부 오류를 사용자 친화 메시지로 매핑

**문제**: 쓰기 실패(RLS 거부, 세션 만료 등) 시 `toast.error(... e.message)`로 PostgREST raw 메시지(`new row violates row-level security policy` 등)가 그대로 노출됨. 사용자는 원인을 알 수 없음.

**변경**
1. `src/lib/authErrors.ts` (신규) — 공통 매퍼 작성:
   ```ts
   // Supabase/PostgREST 오류를 사용자 언어 메시지로 변환
   // 42501 / "row-level security" / 401·403 → 권한/로그인 안내
   export function friendlyAuthError(e: unknown, lang: "ko" | "id"): string
   ```
   - 매핑: 권한거부 → ko `"권한이 없습니다. 관리자에게 문의하세요."` / id `"Anda tidak memiliki izin. Hubungi administrator."`
   - 세션 만료/미인증 → ko `"로그인이 필요합니다."` / id `"Perlu masuk (login)."`
   - 그 외 → 기존 `e.message` 유지.
2. 적용처: `InspectionTab.tsx`의 `handleDelete`, 저장 핸들러, `createVehicle` catch 블록의 `toast.error(...)`를 `friendlyAuthError(e, lang)`로 교체.

**수용 기준**: `user`(읽기) 계정으로 (게이팅을 우회해) 저장/삭제를 시도해도 raw SQL 오류가 아닌 "권한이 없습니다" 토스트가 뜬다. 정상 계정의 성공 흐름은 변화 없음.

---

### [P0-2] 쓰기·다운로드 액션 전수 게이팅 감사(회귀 방지)

**문제**: 신규 탭/버튼 추가 시 게이팅 누락 위험. 현재는 누락 없음(확인 완료)이나 회귀 방지 장치가 없음.

**변경**
1. 아래 쓰기 경로가 **모두** 권한 플래그로 감싸져 있는지 확인하고, 누락 시 게이팅 추가:
   - `src/api/vehicles.ts` → 호출부(`InspectionTab` 차량추가) `canWrite`
   - `src/api/inspections.ts` `saveInspectionRound`(upsert) → 호출부 `canWrite`
   - `src/api/inspections.ts` `deleteInspectionRound`(delete) → 호출부 `canDelete`
   - `src/components/LifeTab.tsx` 엑셀 export → `canDownload`
2. 회귀 방지 주석 규칙: 각 쓰기 API 함수 상단에 `// ⚠ 호출부는 반드시 canWrite/canDelete로 게이팅 + RLS가 최종 방어선` 주석 통일.

**수용 기준**: 저장소 전체 `grep -nE "\.(insert|update|upsert|delete)\("` 결과의 모든 호출부가 권한 플래그 뒤에 있음을 리뷰 코멘트로 명시.

### [P1-1] 헤더에 현재 역할 배지 표시

**문제**: 로그인해도 자신의 권한 등급을 알 수 없어(헤더엔 이메일만) 버튼이 왜 숨는지 사용자가 혼란.

**변경**
1. `src/pages/home/Index.tsx` 헤더의 이메일 표시 옆에 역할 배지 추가. `useAuth()`의 `role` 사용.
2. 배지 색상은 **프로젝트 디자인 팔레트** 준수(연한 배경 + 어두운 청회색 텍스트 `#546E7A`):
   - `admin` → SOFT BLUE 배경 `#E3F2FD`
   - `staff` → SAGE GREEN 배경 `#E8F5E9`
   - `user` → PASTEL BLUE-GREY 배경 `#ECEFF1`
3. 라벨은 i18n: ko `관리자/실무자/조회` · id `Admin/Staf/Lihat`. `src/i18n/`에 키 추가(`role.admin` 등).

**수용 기준**: 로그인 시 이메일 왼쪽에 역할 배지가 뜨고, 역할별 색상·다국어가 올바르게 표시된다. 비로그인 시 배지 없음.

---

### [P1-2] 읽기 전용(user) 사용자 안내 메시지

**문제**: `user` 역할이 입력 탭을 열면 폼이 그냥 숨겨져(`canWrite` false) 왜 입력이 안 되는지 알 수 없음.

**변경**
1. `src/components/InspectionTab.tsx`의 `formOpen && (canWrite ? (...) : (...))` else 분기에, 로그인은 했으나 `!canWrite`인 경우 안내 박스 표시:
   - ko `"조회 권한 계정입니다. 입력·수정은 실무자(staff) 이상 권한이 필요합니다."`
   - id `"Akun hanya-baca. Input memerlukan izin staf ke atas."`
2. 비로그인(`!user`)일 때는 기존 로그인 유도(`onLoginClick`) 유지.

**수용 기준**: user 계정 로그인 상태에서 입력 탭에 안내 문구가 보이고, 로그인 버튼이 아닌 "권한 필요" 안내가 뜬다.

---

### [P1-3] 역할 변경 후 재로그인 필요 안내

**문제**: 관리자가 `app_metadata.role`을 바꿔도 기존 JWT에는 반영 안 됨 → 사용자는 재로그인(또는 토큰 리프레시) 전까지 이전 권한.

**변경**: `docs/사용설명서.md`(또는 운영 노트)에 "역할 변경 후에는 해당 사용자가 **로그아웃 후 재로그인**해야 적용됩니다" 1줄 명시. 코드 변경 불필요.

**수용 기준**: 문서에 반영.

---

### [P2-1] (선택·2단계) 소프트 삭제 + 감사 컬럼

**배경**: 현장(카리만탄) 원격 입력 특성상 오삭제 복구 안전망과 "누가 변경했는지" 추적이 없음.

**변경**
1. `supabase/migrations/0009_inspections_audit_softdelete.sql` (신규):
   - `inspections`에 `deleted_at timestamptz`, `created_by uuid`, `updated_by uuid` 추가.
   - SELECT 정책에 `deleted_at is null` 조건 추가.
   - `deleteInspectionRound`는 **물리 삭제 대신 `deleted_at=now()` 업데이트**(soft delete)로 변경, 물리 삭제(하드)는 admin 전용 별도 경로로 유지.
2. `src/api/inspections.ts` `deleteInspectionRound` 로직 수정.

**수용 기준**: 삭제 후 행이 DB에 남고 화면에서만 사라진다. admin이 복구(=`deleted_at=null`) 가능. **단, 이 과제는 데이터 계약 변경이므로 P0·P1 완료 후 별도 승인 하에 진행.**

### [P2-2] 사용자·역할 관리 절차 문서화 + 셀프 가입 차단 확인

**변경**
1. `docs/권한관리-운영.md` (신규): Supabase 대시보드에서 사용자 초대 → `app_metadata`에 `{"role":"staff"}` 설정 절차를 스크린샷 없이 단계별 서술.
2. Supabase 대시보드 **Authentication → Sign-ups 비활성화 여부 확인** 안내 추가(임의 가입 시 기본 `user`라 읽기만 가능하나, 운영 원칙상 초대제 권장).

**수용 기준**: 문서만으로 신규 인원 권한 부여가 가능. 코드 변경 없음.

---

### [P2-3] 하드코딩 폴백 키 정리

**문제**: `src/lib/supabase.ts`에 URL·publishable 키 하드코딩 폴백 존재. publishable 키는 노출 안전 설계이나, 공개 저장소에 상수로 두는 것은 지양.

**변경**: `.env`(`VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`) 필수화. 폴백 상수 제거하고, 누락 시 명시적 콘솔 경고 + `SUPABASE_ENABLED=false` 처리. `.env.example`는 유지.

**수용 기준**: env 미설정 시 명확한 경고가 뜨고, 하드코딩 키가 소스에서 사라진다.

## 2. 작업 순서 및 커밋 전략

1. **P0-1 → P0-2** (보안/정합성) 먼저. 커밋 단위: 과제별 1커밋.
2. **P1-1 → P1-2 → P1-3** (UX/운영).
3. **P2-*** 는 사용자 승인 후 진행(특히 P2-1은 데이터 계약 변경).
4. 커밋 메시지 예: `fix(auth): map RLS denial to friendly message (P0-1)` / `feat(auth): role badge in header (P1-1)`.
5. 각 커밋 전 `npm run build`(또는 `tsc`) 통과 확인.

## 3. 검증 체크리스트 (완료 판정)

- [ ] admin 계정: 입력·수정·삭제·엑셀 다운로드 모두 동작.
- [ ] staff 계정: 입력·수정·다운로드 동작, **삭제 버튼 미노출**, (우회 시도 시) 친화 오류.
- [ ] user 계정: 조회만 가능, 입력 탭에 "권한 필요" 안내, 저장/삭제 우회 시 "권한이 없습니다".
- [ ] 비로그인: 조회만, 입력 탭에 로그인 유도.
- [ ] 헤더 역할 배지가 역할·언어별로 정확.
- [ ] `npm run build` 무오류.
- [ ] `grep`로 모든 쓰기 호출부 게이팅 확인.

## 4. 절대 하지 말 것 (가드레일)

- RLS 정책(`0006`,`0007`)을 되돌리거나 `for all using(true)` 형태로 완화하지 말 것.
- 서비스 롤 키(`service_role`)를 클라이언트 코드에 넣지 말 것.
- 역할 판정을 프런트에서만 하고 RLS를 제거하지 말 것 — **RLS가 최종 방어선**.
- 데이터 스키마 변경(P2-1)은 명시적 승인 전 착수 금지.
