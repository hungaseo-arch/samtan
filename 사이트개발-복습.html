차량 배치도 가운데 정렬 + 크기 1.5배

파일: src/components/LayoutTab.tsx. 두 요구사항은 서로 다른 원리.

1. 가운데 정렬

좌측에 붙는 이유: 92행 차량 줄의 min-w-max(콘텐츠 폭만큼만 차지) 때문. 화면보다 좁으면 왼쪽 고정.

  • 해결: min-w-max → w-max mx-auto(콘텐츠 폭 + 좌우 자동 여백)로 변경.
  • 화면보다 좁으면 가운데 정렬, 넓으면 기존 가로 스크롤 유지.

2. 크기 1.5배

문제: 타이어·간격이 고정 Tailwind 값으로 흩어져 있음 — TireCell w-10 h-12(238·258행), 비시험 w-8 h-10(245행), 축 라벨 w-5 h-5(115행), 간격 gap-1·gap-3, 패딩 px-3 pt-5 pb-3(97행). 일일이 1.5배는 번거로움.

방법내용평가
A. zoom: 1.592행에 style={{ zoom: 1.5 }}권장 — 한 줄로 전체 확대, 공간도 함께 늘어 범례 안 겹침
B. transform: scale(1.5)같은 위치 transform시각만 커지고 공간 미확보 → 겹침 위험
C. 유틸 일괄 1.5배w-10→w-[60px] 등정석이나 손 많이 감
복습 포인트: 정렬·크기를 92행 한 줄에서 동시 처리 가능 — className의 min-w-max를 w-max mx-auto로, 그리고 style={{ zoom: 1.5 }} 추가. Chrome 기준 사내 도구라 zoom 방식이 가장 깔끔.

질문 4

대시보드 '주의·교체 알림' 1열 → 2열

파일: src/components/DashboardTab.tsx 75행.

원리

현재 1열인 이유: <ul className="space-y-2">. space-y-2는 자식 사이에 세로 간격만 줘서 위→아래로 쌓는 방식.

  • 2열로: 레이아웃을 CSS 그리드로 전환 → grid grid-cols-1 sm:grid-cols-2 gap-2.
  • sm: = 모바일 1열, 일정 폭 이상 2열 (반응형).
  • 그리드에선 space-y-2 대신 gap-2 사용. 각 <li> 스타일(76~83행)은 그대로.
복습 포인트: "쌓기(space-y)"는 세로 1열, "그리드(grid-cols-N)"는 N열 배치. 항목 스타일은 건드릴 필요 없이 부모 ul의 클래스만 바꾸면 된다.
VS Code Claude에게 줄 지시 예시

  2. 변경 후 npm run lint 및 브라우저에서 2열 배치·반응형(좁은 화면 1열) 확인.

질문 7

메뉴 탭 가운데 정렬

파일: src/pages/home/Index.tsx 105행 <nav>.

원리

<nav className="flex gap-0 overflow-x-auto"> — nav는 전체 폭 flex 행이고 자식(탭 버튼)은 기본 justify-start라 좌측에 붙음. justify-center 추가로 가운데 정렬.

  • 변경: <nav className="flex justify-center gap-0 overflow-x-auto">
복습 포인트: flex 안에서 가로 정렬은 justify-*(start/center/end/between)로 제어. 세로 정렬은 items-*.
주의(반응형): overflow-x-auto + justify-center는 좁은 화면에서 탭이 넘칠 때 왼쪽 끝이 스크롤로 안 닿을 수 있음. 모바일까지 안전하게는 justify-start sm:justify-center(일정 폭 이상에서만 가운데).

  2. 모바일 안전성까지 원하면 justify-start sm:justify-center로 적용.
  3. 변경 후 npm run lint 및 데스크톱·모바일 폭에서 정렬·스크롤 확인.

클래스결과
justify-between첫 항목 왼쪽 끝, 마지막 오른쪽 끝, 사이 균등

질문 10

Lifetime DB: 수명(km) + 수명(hr) 동시 표시 + 예상 KM 추가

파일: src/components/LifeTab.tsx(표시) + src/data/tmsUtils.ts(계산). 단순 스타일이 아닌 데이터 계산 + 모델 설계 작업.

확정 결정

  • 예상수명 임계값: 5mm(교체권장, TREAD_DANGER).
  • 수명 기준점: 점검 회차별 상세 데이터(units의 dates/hm/km + tread 시계열).

핵심 문제: 데이터 연결

Lifetime DB는 시리얼 70본 표지만, 트레드·km 추세는 모니터링 5대×10포지션(50본)에만 tread[ch][pos]·units[].km로 존재. 따라서 시리얼→현재 ch/pos 매핑(serial2 || serial)되는 타이어만 값이 나오고 나머지는 "−".

데이터 현실: 실제 km 기록은 CH 834·839뿐. CH 832·835는 HM Reset·km 결측, CH 849는 km=0. 그래서 수명(km)·예상 KM은 일부만, 수명(hr)은 대부분 계산됨. 코드 문제 아님 — km 입력이 채워지면 자동 표시.

계산 원리

  • 수명(km) = 기록 회차 (최신 km − 최초 km).
  • 수명(hr) = (최신 HM − 최초 HM).
  • 예상 KM = 트레드 vs km 선형회귀(추세선)로 기울기(mm/km) 산출 → 트레드가 5mm 도달하는 누적 km 외삽. 회귀식 slope = Σ((x−x̄)(y−ȳ)) / Σ((x−x̄)²), 예상km = (5 − intercept) / slope.
복습 포인트: 계산 로직은 tmsUtils.ts에 lifeCalc(serial) 헬퍼로 분리(README "계산 분리" 원칙) → 표에서 호출만. 추세 예측 = 선형회귀 기울기 + 임계값 외삽.
정확도 개선 여지: 교체 이력(TMS_DATA.repl) 있는 포지션은 신·구 타이어 혼합이라, 정밀하게는 교체일 이후 데이터만 회귀에 사용. v1은 전체 구간, 후속 개선 권장.
